# Stage: Identify Boundaries

## Intent

Select decision boundaries worth documenting from the structural map.

## Category and Posture

**Category**: Extract
**Posture**: Index, don't analyse. Identify where decisions happen and what type they are. Do not extract contracts, scenarios, or detailed behaviour descriptions — that's Stage 4's work.

## Input

Read the Codebase Structural Map from `autodoc-workspace/structural-map.md`.
Read `autodoc/contracts/codebase-structural-map.md` for the input schema.

Read the Reference Index from `autodoc-workspace/reference-index.md` (passthrough artifact from Stage 1).
Read `autodoc/contracts/reference-index.md` for the schema. If the Reference Index is empty, proceed with heuristics alone.

## Output

Write the Boundary List to `autodoc-workspace/boundary-list.md`.

Read `autodoc/contracts/boundary-list.md` for the output schema.

## Steps

1. Load the Codebase Structural Map
2. Load the Reference Index (may be empty)
3. If the Reference Index has entries, extract the topic list to use as sensitivity bias — be more attentive to code regions related to reference topics
4. Scan the structural map construct by construct. For each construct:
   a. Read the source code at the construct's file and line range using the Read tool
   b. Evaluate whether the code contains a decision boundary using the heuristics below
   c. If a boundary is found, record it with location, type, decision summary, and verbatim evidence
   d. If the Reference Index covers topics related to this boundary, record the matching `source_locator` values in `reference_hints`
5. After scanning all constructs, review the boundary list for completeness
6. Assemble the Boundary List

## Guidance

### Inclusion Heuristics

Include a boundary when the code exhibits one of these patterns:

1. **Conditional branching with side effects** — `if/else`, `switch/match` blocks that change system state, call external services, or return different response types
2. **Error handling boundaries** — try/catch, error return patterns where the system chooses between recovery, retry, propagation, or fallback
3. **Validation gates** — input validation, authorization checks, precondition assertions that accept or reject
4. **State transitions** — code that moves an entity from one status to another (e.g., `pending` → `charged`, `draft` → `published`)
5. **Integration points** — calls to external services, databases, message queues where the system must handle success/failure/timeout
6. **Configuration-driven behaviour** — feature flags, environment-dependent logic, strategy patterns

### Exclusion Criteria

Do **not** include:

1. Pure data access with no branching (getters, setters, simple queries)
2. Pass-through wrappers that delegate without adding decisions
3. Boilerplate code generated by frameworks
4. Trivial null/nil checks that are language idioms rather than domain decisions

### Decision Summary Writing

Each `decision_summary` must be one sentence with a single decision verb. Good: "Determines whether to retry a failed payment or mark it as permanently failed." Bad: "Handles payment retries and error logging." (Two verbs, vague.)

### Evidence Rules

The `evidence` field must contain verbatim source code — the actual branching point, max 10 lines. Do not paraphrase or summarize the code.

### Reference Sensitivity

When the Reference Index is non-empty, use its topics as a bias signal. If references discuss "retry policy", be more attentive to retry-related conditional branches. This adjusts heuristic sensitivity — it does not add new heuristic categories or override exclusion criteria.

- **Do not** force-include boundaries just because they relate to a reference topic
- **Do not** skip boundaries just because no reference covers them
- **Do** record which reference `source_locator` values informed the inclusion in `reference_hints`
