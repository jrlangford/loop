# Stage Decomposition

## Pipeline Overview
Extract structured behaviour documents from source code by surveying the codebase, identifying decision boundaries, and producing one behaviour document per boundary with contracts, scenarios, and traceability to code and tests.

Total stages: 5

## Stages

### Stage 1: Gather References
- **Category**: Enrich
- **Intent**: Load optional reference documents into a structured reference index
- **Input**: User-provided list of reference paths (local files, URLs, Notion page IDs); may be empty
- **Output**: Reference index — a structured collection of reference content with source attribution, keyed by topic/domain area for downstream lookup
- **Sources**:
  - Local filesystem (markdown, text, HTML files)
  - URLs (fetched via web)
  - Notion pages (via Notion MCP — requires MCP server installed and configured)
- **Sinks**: None
- **Complexity**: Source diversity — each reference type requires different retrieval. Notion MCP availability must be checked as a precondition when Notion references are specified. When no references are provided, this stage produces an empty index and downstream stages proceed without it.

### Stage 2: Survey Codebase
- **Category**: Extract
- **Intent**: Build a structural map of the target code
- **Input**: File/directory paths specified by user (or entire repository root)
- **Output**: Codebase structural map — files, functions/methods, classes/modules, call relationships, and test file locations mapped to their corresponding source files
- **Sources**: Local filesystem (source code files)
- **Sinks**: None
- **Complexity**: Language-agnostic — must adapt structural extraction to whatever language(s) are present. Must identify test files and map them to the source files they cover using language-specific conventions (e.g., `*_test.go`, `test_*.py`, `*.spec.ts`, `__tests__/`). Large repos may exceed a single context window — the structural map must be a compressed representation, not raw file contents.

### Stage 3: Identify Boundaries
- **Category**: Extract
- **Intent**: Select decision boundaries worth documenting from the structural map
- **Input**: Codebase structural map (from Stage 2), source code files
- **Output**: Boundary list — each entry contains: location (file:function), boundary type, brief description of the decision being made, and pointers to the relevant code region
- **Sources**: Local filesystem (reading source code at locations identified in the structural map)
- **Sinks**: None
- **Complexity**: This is the core judgment stage. Applies heuristics to distinguish genuine decision boundaries from trivial logic:
  1. **Conditional branching with side effects** — if/else, switch/match that changes state, calls external services, or returns different response types
  2. **Error handling boundaries** — try/catch, error returns where the system chooses recovery, retry, propagation, or fallback
  3. **Validation gates** — input validation, authorization checks, precondition assertions that accept or reject
  4. **State transitions** — code that moves an entity between statuses
  5. **Integration points** — calls to external services, databases, message queues with success/failure/timeout handling
  6. **Configuration-driven behaviour** — feature flags, environment-dependent logic, strategy patterns

  **Exclusion criteria** (not worth documenting):
  - Pure data access with no branching (getters, setters, simple queries)
  - Pass-through wrappers that delegate without adding decisions
  - Boilerplate code generated by frameworks
  - Trivial null/nil checks that are language idioms rather than domain decisions

### Stage 4: Extract Behaviours
- **Category**: Transform
- **Intent**: Produce one behaviour document per decision boundary
- **Input**: Boundary list (from Stage 3), source code files, reference index (from Stage 1, may be empty), behaviour template
- **Output**: Set of behaviour documents — one markdown file per boundary, following `behaviour-template.md` format, with filled-in contracts (pre/post/invariants), scenarios, and traceability
- **Sources**: Local filesystem (reading source code and test files at locations identified in the boundary list)
- **Sinks**: None
- **Complexity**: This is the fan-out point — each boundary can be extracted independently. For each boundary, the stage:
  1. Reads the code region around the decision boundary
  2. Extracts pre/postconditions and invariants from the code logic
  3. Generates at least one Given/When/Then scenario (happy path required; edge cases best-effort)
  4. Locates test files covering this boundary using the test file mapping from Stage 2 — searches for test functions referencing the boundary's function/method name
  5. Uses existing test assertions as evidence to inform contract extraction (test setup → preconditions, assertions → postconditions)
  6. Records code references (file:function) and test references in the traceability section
  7. When a pre/postcondition is relevant but not testable, states this explicitly with rationale
  8. Assigns a non-sequential ID (`BHV-<hex>`)

### Stage 5: Deduplicate Behaviours
- **Category**: Evaluate
- **Intent**: Detect and merge behaviour documents covering the same decision
- **Input**: Set of behaviour documents (from Stage 4)
- **Output**: Deduplicated set of behaviour documents — duplicates merged, traceability links combined, final count reported
- **Sources**: None
- **Sinks**: None
- **Complexity**: Duplication can arise when the same logical decision is implemented across multiple code paths (e.g., validation repeated in a handler and a middleware). The stage must distinguish true duplicates (same decision, different locations — merge and combine traceability) from similar-looking but distinct behaviours (same pattern, different domain meaning — keep both). When merging, the richer document is preserved and traceability links from both are combined.
